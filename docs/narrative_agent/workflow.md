# Workflow

## Runtime flow

Current CLI flow in `main.py`:

1. Load input from JSON or interactive prompts.
2. Invoke the compiled LangGraph app once.
3. Print generated Markdown.
4. Loop on terminal commands:
   - `f`: revise with feedback
   - `s`: save to `outputs/...`
   - `q`: exit without saving

## Current graph shape

Defined in `src/agente_narrativa/grafo.py`:

```text
gerar_historia -> END
atualizar_historia -> END
salvar_historia -> END
```

Important detail:

- Only `gerar_historia` is used as the graph entry point.
- `atualizar_historia` and `salvar_historia` are called directly from `main.py`, not traversed through the compiled graph.
- `MemorySaver` is configured, but the effective interaction model is still mostly CLI-managed state passing.

## Node roles

### `gerar_historia`

- Builds the initial user prompt from the structured input.
- Sends `SystemMessage + HumanMessage` to the shared LLM.
- Returns:
  - generated Markdown in `historia`
  - message history in `historico`
  - empty `feedback`

### `atualizar_historia`

- Takes the full prior message history plus a feedback message.
- Asks the LLM to revise the full story while preserving the method.
- Replaces `historia` with the revised Markdown.
- Appends the new exchange to `historico`.

### `salvar_historia`

- Slugifies `empresa` and `dificuldade`.
- Creates `outputs/<Empresa>/<Empresa>_<dificuldade>_<nnn>/`.
- Pre-creates `documentos/documentos_gerados/`.
- Saves the Markdown file as `<Empresa>_<dificuldade>_<nnn>.md`.

## Persistence behavior

- IDs are generated by counting existing directories matching the pattern.
- The next case number is `len(existentes) + 1`, so numbering is filesystem-dependent.
- The same hardcoded thread id `sessao_1` is always passed to the graph config in `main.py`.

## Migration implication

If the new architecture introduces real graph-driven checkpoints or resumability, keep the user-visible flow equivalent:

- generate
- optionally revise one or more times
- save only when requested
